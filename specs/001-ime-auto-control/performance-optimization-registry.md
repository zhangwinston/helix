# IME Registry 全局互斥锁性能优化方案

## 问题分析

### 当前实现的问题

1. **锁内执行系统 API 调用**
   - 在 `handle_cursor_move` 的最后一次 `registry::with_context_mut` 中，锁内调用了 `set_ime_enabled(target)`
   - 系统 API 调用可能阻塞（Windows IME API、macOS Input Method API 等）
   - 导致所有 document+view 的操作被串行化

2. **多次获取全局锁**
   - `handle_cursor_move` 中至少 4-5 次获取全局锁
   - 每次锁获取都会产生竞争，即使操作很快也会累积延迟

3. **全局锁串行化所有操作**
   - 所有 document+view 组合共享一个全局 `Mutex<ImeRegistry>`
   - 即使不同 document/view 的操作也会互相阻塞

## 优化方案

### 方案 1: 将系统 API 调用移到锁外（推荐，立即实施）

**核心思路**：在锁内只做状态决策和缓存更新，系统 API 调用在锁外执行。

**实现步骤**：
1. 修改 `with_context_mut` 的调用模式，在锁内只返回需要执行的操作
2. 在锁外执行 `set_ime_enabled` 和 `read_ime_enabled`
3. 系统 API 调用成功后，再次获取锁更新缓存

**优点**：
- 实现简单，风险低
- 立即减少锁持有时间
- 不影响现有架构

**缺点**：
- 仍使用全局锁，不同 document/view 仍可能竞争
- 需要两次锁获取来更新缓存（但第二次通常很快）

### 方案 2: 使用细粒度锁（中期优化）

**核心思路**：为每个 (doc_id, view_id) 组合使用独立的锁，而不是全局锁。

**实现步骤**：
1. 使用 `DashMap<(DocumentId, ViewId), Mutex<ImeContext>>` 替代 `Mutex<HashMap<...>>`
2. 每个 context 独立加锁，互不干扰
3. 清理操作（prune_orphans）需要遍历所有条目，但频率低

**优点**：
- 不同 document/view 的操作完全并行
- 锁竞争大幅减少
- 适合多文档/多视图场景

**缺点**：
- 实现复杂度较高
- 需要处理 DashMap 的并发清理逻辑
- 内存开销略增（每个条目一个 Mutex）

### 方案 3: 读写锁优化（可选）

**核心思路**：如果读操作远多于写操作，使用 `RwLock` 替代 `Mutex`。

**实现步骤**：
1. 将 `Mutex<ImeRegistry>` 改为 `RwLock<ImeRegistry>`
2. 读操作（如 `context()`）使用读锁
3. 写操作（如 `with_context_mut`）使用写锁

**优点**：
- 多个读操作可以并行
- 适合读多写少的场景

**缺点**：
- 写操作仍会阻塞所有读操作
- 如果写操作频繁，性能提升有限
- Rust 的 `RwLock` 在某些场景下可能不如 `Mutex` 高效

### 方案 4: 无锁数据结构（长期优化，可选）

**核心思路**：使用原子操作和 `Arc` 实现无锁的 context 访问。

**实现步骤**：
1. 使用 `ArcSwap` 或 `Arc<AtomicPtr<...>>` 存储 contexts
2. 使用 CAS 操作更新 context
3. 需要处理内存回收问题

**优点**：
- 理论上性能最优
- 无锁竞争

**缺点**：
- 实现复杂度很高
- 需要处理 ABA 问题
- 可能过度设计

## 推荐实施路径

### 阶段 1: 立即实施（方案 1）
- **目标**：消除锁内系统 API 调用
- **工作量**：1-2 小时
- **风险**：低
- **收益**：显著减少锁持有时间，避免系统 API 阻塞导致的串行化

### 阶段 2: 中期优化（方案 2）
- **目标**：实现细粒度锁，消除不同 document/view 间的竞争
- **工作量**：4-6 小时
- **风险**：中
- **收益**：多文档/多视图场景下性能大幅提升

### 阶段 3: 评估是否需要（方案 3/4）
- **目标**：根据实际性能数据决定是否需要进一步优化
- **条件**：如果方案 1+2 已满足性能要求，可跳过

## 性能预期

### 方案 1 预期效果
- 锁持有时间：从 ~10-50ms（含系统 API）降至 ~0.1-1ms
- 锁竞争：减少 80-90%
- 多文档场景：仍有一定竞争，但不会因系统 API 阻塞

### 方案 2 预期效果
- 锁竞争：不同 document/view 完全消除竞争
- 并发性能：多文档场景下接近线性扩展
- 单文档场景：与方案 1 相当

## 实施注意事项

1. **保持向后兼容**：优化不应改变 API 语义
2. **错误处理**：系统 API 调用失败时的缓存更新逻辑
3. **测试覆盖**：确保多文档/多视图场景的测试覆盖
4. **性能监控**：添加锁竞争指标，验证优化效果
