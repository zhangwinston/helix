# 退出模式强制查询对性能优化的影响分析

## 改动对比

### 光标移动（高频操作）- 保持不变

```rust
// 仍然使用缓存优化
let cached_ime_state = registry::with_context_mut(..., |ctx| ctx.cached_ime_state);
let current_ime_enabled = cached_ime_state.unwrap_or_else(|| read_ime_enabled("cursor move"));
```

**特点**：
- ✅ 优先使用缓存（快速路径）
- ✅ 缓存未命中时才查询系统
- ✅ 系统 API 调用在锁外执行

### 退出 Insert 模式（低频操作）- 已修改

**修改前**：
```rust
let cached_ime_state = registry::with_context_mut(..., |ctx| ctx.cached_ime_state);
let current_ime_enabled = cached_ime_state.unwrap_or_else(|| read_ime_enabled("mode switch"));
```

**修改后**：
```rust
// 强制查询系统状态
let current_ime_enabled = read_ime_enabled("mode switch exit");
```

## 性能影响分析

### ✅ 不影响核心性能优化原则

1. **系统 API 调用仍在锁外**
   - `read_ime_enabled` 在锁外执行
   - 不会阻塞其他 document/view 的操作
   - 锁内只做快速的内存操作（状态决策、缓存更新）

2. **高频操作仍使用缓存**
   - 光标移动（最频繁的操作）仍然优先使用缓存
   - 缓存命中时完全避免系统 API 调用
   - 性能优化的主要收益保持不变

3. **低频操作的影响可忽略**
   - 退出 Insert 模式是用户主动操作，频率远低于光标移动
   - 即使每次退出都查询系统，对整体性能影响微乎其微
   - 用户感知不到延迟（退出模式本身就是"慢操作"）

### 性能数据估算

**假设场景**：
- 光标移动：1000 次/分钟（高频）
- 退出 Insert 模式：10 次/分钟（低频）

**缓存命中率**（光标移动）：
- 假设 90% 缓存命中率
- 1000 × 0.9 = 900 次避免系统调用
- 1000 × 0.1 = 100 次系统调用

**退出模式**：
- 10 次系统调用（强制查询）

**总系统调用**：
- 修改前：100 + 10 = 110 次/分钟
- 修改后：100 + 10 = 110 次/分钟（相同）

**结论**：如果退出模式时缓存命中，修改后会增加 10 次系统调用，但：
- 频率低（10 次/分钟）
- 用户感知不到（退出模式本身需要时间）
- 保证了功能正确性（避免缓存不一致导致的 bug）

## 性能优化原则保持

### ✅ 方案 1 的核心优化仍然有效

1. **锁内不执行系统 API**
   - ✅ 所有系统 API 调用（`read_ime_enabled`, `set_ime_enabled`）都在锁外
   - ✅ 锁持有时间仍然很短（<1ms）

2. **减少锁竞争**
   - ✅ 系统 API 调用不会阻塞其他操作
   - ✅ 不同 document/view 的操作可以并行

3. **缓存优化**
   - ✅ 光标移动（高频）仍然使用缓存
   - ✅ 退出模式（低频）强制查询，影响可忽略

## 权衡分析

### 性能 vs 正确性

| 方案 | 性能 | 正确性 | 推荐度 |
|------|------|--------|--------|
| 使用缓存 | 最优 | 可能不准确 | ⚠️ |
| 强制查询 | 略差（可忽略） | 准确 | ✅ |

**结论**：为了功能正确性，在低频操作（退出模式）中强制查询是合理的权衡。

## 进一步优化建议（可选）

如果未来需要进一步优化，可以考虑：

1. **混合策略**
   - 退出模式时：先检查缓存，如果缓存存在且最近更新过（<1秒），使用缓存
   - 否则强制查询系统

2. **异步查询**
   - 退出模式时异步查询系统状态
   - 不阻塞用户操作

3. **状态监听**
   - 监听系统 IME 状态变化事件（如果平台支持）
   - 自动同步缓存

但这些优化在当前场景下**不是必需的**，因为退出模式是低频操作，影响可忽略。

## 总结

✅ **改动不影响性能优化的核心原则**
- 系统 API 调用仍在锁外
- 高频操作（光标移动）仍使用缓存
- 低频操作（退出模式）强制查询，影响可忽略

✅ **保证了功能正确性**
- 避免了缓存不一致导致的 bug
- 确保退出模式时 IME 正确关闭

✅ **性能影响可忽略**
- 退出模式频率低（用户主动操作）
- 用户感知不到延迟
- 整体性能优化收益保持不变
